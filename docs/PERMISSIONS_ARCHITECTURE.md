## ğŸ”„ Permission Flow & Merging

### How Permissions are Combined

```yaml
# Example: Job "analytics_pipeline" in "prod" target

# 1. Bundle-level (databricks.yml)
bundle:
  permissions:
    - level: CAN_VIEW
      group_name: security_auditors

# 2. Target-level (databricks.yml)
targets:
  prod:
    permissions:
      - level: CAN_VIEW
        group_name: data_team
      - level: CAN_RUN
        group_name: operations

# 3. Resource-level (resources/analytics_pipeline.job.yml)
resources:
  jobs:
    analytics_pipeline:
      permissions:
        - level: CAN_MANAGE
          user_name: analytics-lead@company.com

# Result: All three are MERGED (not overridden)
# Final permissions for analytics_pipeline in prod:
#   - security_auditors: CAN_VIEW
#   - data_team: CAN_VIEW
#   - operations: CAN_RUN
#   - analytics-lead@company.com: CAN_MANAGE
```

### 1. **Development Mode vs Production Mode**

```yaml
# Development Mode (user target)
mode: development
# â€¢ Automatic full access for deploying user
# â€¢ Resources prefixed with [user <name>]
# â€¢ No explicit permissions needed
# â€¢ Fast iteration

# Production Mode (dev, stage, prod)
mode: production
# â€¢ Explicit permissions required
# â€¢ Resources shared across team
# â€¢ Service principals recommended
# â€¢ Controlled access
```

### 2. **run_as vs permissions**

```yaml
# run_as: WHO executes the jobs/workflows
run_as:
  service_principal_name: <sp-id>
# â†’ All jobs in this target run as this SP

# permissions: WHO can view/manage/run the resources
permissions:
  - level: CAN_MANAGE
    service_principal_name: <sp-id>
# â†’ This SP can manage the bundle resources
```

**Best Practice:** Use the same SP for both in production:
```yaml
targets:
  prod:
    run_as:
      service_principal_name: ${var.prod_sp}
    permissions:
      - level: CAN_MANAGE
        service_principal_name: ${var.prod_sp}
```

### 3. **Permission Inheritance**

Permissions do NOT inherit in a hierarchical way. They MERGE:

```yaml
# This is how it works:
CAN_VIEW + CAN_RUN = Both permissions granted (not CAN_RUN only)

# NOT like this:
CAN_MANAGE â‰  Automatically includes CAN_VIEW
# (You need to grant CAN_MANAGE explicitly)
```
